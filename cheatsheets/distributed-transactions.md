# 分布式事务速查（🎯 案例 + ✍️ 论文高频）

## 一、CAP 与 BASE 基础

### CAP 三选二

- **C** Consistency：一致性
- **A** Availability：可用性
- **P** Partition Tolerance：分区容忍性

> 分布式系统**必须 P**，故实际在 **CP** 与 **AP** 之间选

### BASE 理论

- **Basically Available**（基本可用）
- **Soft state**（软状态）
- **Eventually consistent**（最终一致性）

## 二、五大分布式事务方案对比

| 方案 | 一致性 | 性能 | 复杂度 | 典型场景 |
|---|---|---|---|---|
| **2PC** | 强 | 低 | 低 | 跨库事务、XA |
| **3PC** | 强 | 更低 | 中 | 理论优化 2PC |
| **TCC** | 强 | 中 | **高** | 金融核心交易 |
| **本地消息表** | 最终 | 高 | 低 | 电商订单 |
| **Saga** | 最终 | 高 | 中 | 长流程业务 |
| **MQ 事务消息** | 最终 | 高 | 低 | RocketMQ 场景 |

## 三、2PC（Two-Phase Commit）

### 阶段

1. **准备阶段**：协调者问"能提交吗？"所有参与者回"Yes/No"
2. **提交阶段**：全 Yes → Commit；否则 Rollback

### 问题

- **同步阻塞**：参与者锁资源等协调者
- **协调者单点**
- **数据不一致**：阶段 2 部分参与者收到 Commit，其余未收到

## 四、3PC（Three-Phase Commit）

在 2PC 基础上加 **CanCommit** 阶段 + **超时机制**，减少阻塞

- CanCommit → PreCommit → DoCommit
- 仍不能完全解决一致性问题

## 五、TCC（Try-Confirm-Cancel）

### 三阶段

1. **Try**：预留业务资源（冻结）
2. **Confirm**：确认提交（幂等）
3. **Cancel**：回滚释放（幂等）

### 优点 / 缺点

✅ 业务可控、性能优于 2PC
❌ 代码侵入大，每个接口写 3 个方法

### 注意事项（**必考**）

- **幂等性**：Confirm/Cancel 可能重复调用
- **空回滚**：未 Try 直接 Cancel 要兼容
- **悬挂**：Cancel 执行后 Try 迟到要拒绝

## 六、Saga 模式

### 两种实现

| 方式 | 说明 |
|---|---|
| **编排式**（Orchestration） | 中心协调者驱动（如 Netflix Conductor） |
| **编制式**（Choreography） | 事件驱动，各服务自治 |

### 特点

- 每步一个本地事务 + 补偿事务
- **不保证隔离性**（中间状态可见）
- 适合长流程（如订单履约）

## 七、本地消息表

### 流程

1. 本地事务：业务 + 消息表**同库写入**
2. 定时任务扫消息表 → 发送 MQ
3. 下游消费 → 回写确认

### 优点

✅ 简单、可靠（借助本地事务保证）

### 缺点

❌ 耦合业务库、定时扫表延迟

## 八、RocketMQ 事务消息

### 流程

1. **Half 消息**：发送到 MQ，对消费者不可见
2. **本地事务**：执行业务
3. **Commit/Rollback**：通知 MQ 提交或回滚
4. **回查**：超时未收到指令，MQ 主动回查业务状态

## 九、幂等性设计（必考）

| 方式 | 说明 |
|---|---|
| **唯一索引** | DB 层防重复 |
| **Token 机制** | 请求前申请 Token |
| **状态机** | 只允许特定状态转移 |
| **分布式锁** | Redis/ZK |
| **乐观锁** | version 字段 |

## 十、一致性协议

| 协议 | 一致性 | 代表系统 |
|---|---|---|
| **Paxos** | 强 | Chubby、ZK（ZAB 近似） |
| **Raft** | 强 | etcd、Consul、TiKV |
| **ZAB** | 强 | ZooKeeper |
| **Gossip** | 最终 | Cassandra、Redis Cluster |

## 十一、选型决策模板（答题用）

| 业务类型 | 推荐方案 | 理由 |
|---|---|---|
| 金融核心 | TCC | 强一致 + 性能可接受 |
| 跨库事务（少量） | 2PC / XA | 简单 |
| 电商订单 | 本地消息表 / MQ 事务 | 最终一致即可 |
| 长流程履约 | Saga | 易扩展 |
| 高并发下单 | **柔性事务**（MQ + 幂等） | 性能优先 |
