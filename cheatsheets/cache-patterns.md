# 缓存模式速查（🎯 案例 + ✍️ 论文高频）

## 一、缓存三大经典问题

### 1. 缓存穿透

**现象**：查询**不存在**的数据，每次都穿透到 DB

**解决**：

- **布隆过滤器**（Bloom Filter）：提前过滤不存在 key
- **空值缓存**：对不存在的 key 也缓存（短 TTL）
- **参数校验**：拦截明显非法请求

### 2. 缓存击穿

**现象**：**某热点 key** 过期瞬间，大量请求同时打 DB

**解决**：

- **互斥锁**（分布式锁）：仅一个线程回源
- **热点 key 永不过期**（后台定时刷新）
- **逻辑过期**：value 带过期时间，过期异步刷新

### 3. 缓存雪崩

**现象**：**大量 key 同时过期**，或缓存服务宕机

**解决**：

- **过期时间加随机值**（避免同时过期）
- **多级缓存**（本地 Caffeine + 分布式 Redis）
- **熔断限流**
- **集群高可用**（主从 + 哨兵 / Cluster）

## 二、缓存一致性策略（经典四种）

| 模式 | 流程 | 优缺点 |
|---|---|---|
| **Cache Aside**（旁路，最常用） | 读：先查缓存，未命中查 DB 并回填<br>写：**先写 DB，再删缓存** | 简单；短暂不一致 |
| **Read/Write Through** | 由缓存层统一管理 DB | 一致性好；实现复杂 |
| **Write Behind / Write Back** | 写缓存立即返回，异步刷 DB | 性能好；可能丢数据 |
| **Refresh Ahead** | 缓存过期前主动刷新 | 命中率高；需预测热点 |

## 三、Cache Aside 的细节陷阱（**必问**）

### 为什么是"先写 DB，再删缓存"而不是"先删缓存，再写 DB"？

- **先删缓存 → 并发读会把旧值重新加载到缓存**
- **先写 DB 再删缓存** 只有极小概率不一致

### 为什么是"删"而不是"更新"？

- 删是幂等的
- 更新需要复杂计算，可能写入不一致的中间值

### 延迟双删（高并发场景）

```
1. 删缓存
2. 写 DB
3. 延迟 N ms 再删一次缓存
```

## 四、热点 key 问题

### 识别

- 监控 QPS Top key
- 业务预判（双 11、秒杀）

### 处理

- **本地缓存**（Caffeine）兜底，减少 Redis 压力
- **多副本分片**：key → key:1 / key:2 / key:3 轮询
- **流量隔离**：热点 key 单独部署

## 五、大 key / 热 key 危害

| 问题 | 危害 |
|---|---|
| **大 key** | 单 key > 10MB 导致网络阻塞、扩容迁移卡顿 |
| **热 key** | 单节点 QPS 过高打爆 CPU |

### 解决

- 大 key 拆分（Hash 拆成多个 key）
- 热 key 多副本分片

## 六、Redis 数据结构选型

| 场景 | 结构 | 命令示例 |
|---|---|---|
| 计数器 | String | INCR |

## 七、分布式锁（Redis 实现）

### 基础命令

```
SET lock_key unique_value NX PX 30000
```

**要点**：

1. **NX**：不存在才设置
2. **PX**：过期时间（防死锁）
3. **unique_value**：释放锁时校验（防误删）

### Redisson 高级特性

- **看门狗**（Watch Dog）自动续期
- 可重入
- 红锁（RedLock）跨多节点

### 分布式锁失效场景（必考）

1. **主从切换**：主挂了，锁还没同步到从 → RedLock 解决
2. **业务执行超时**：锁已过期 → 看门狗续期
3. **GC 停顿**：JVM 长 GC 导致锁过期 → fencing token

## 八、缓存 vs DB 一致性方案对比（答题用）

| 方案 | 一致性 | 复杂度 | 场景 |
|---|---|---|---|
| Cache Aside | 最终 | 低 | **通用首选** |
| 延迟双删 | 最终（更强） | 中 | 高并发 |
| Binlog 订阅（Canal） | 最终（近实时） | 高 | 严格场景 |
| Write Through | 强 | 高 | 一致性要求极高 |
