# 案例分析答题套路集（🎯 决定成败的核心）

## 总原则

- **5 道题选 3 题**：不擅长的直接跳过
- **时间分配**：90 分钟 / 3 题 = 30 分钟 / 题
- **答题顺序**：先答拿手的 → 题干"挖"关键词 → 分点作答
- **字数**：每题总分 25 分，答案每小问 **3–5 句话**，**点多分多**

## 一、通用答题结构

```
1. 一句话复述场景（表明理解题意）
2. 分条作答（1)(2)(3)…）
3. 每条先"是什么"，再"为什么"/"怎么做"
4. 必要时画图 / 列表
```

## 二、题型 1：架构风格选择题

> 典型问法：某系统应采用什么架构风格？理由？

**套路**：

1. **识别业务特征**（高并发、实时、数据流、批处理…）
2. **选择风格**（管道—过滤器 / 事件驱动 / 分层 / 微服务 / 黑板…）
3. **给 3 条理由**：扣住该风格的**核心优势** + **业务匹配度**
4. **指出代价**（权衡点）

### 常见风格速答

| 业务特征 | 推荐风格 | 理由关键词 |
|---|---|---|
| 订单—支付—履约 | 事件驱动 / SOA / 微服务 | 解耦、异步 |
| 流处理 / ETL | 管道—过滤器 | 流水化、独立可替换 |
| 管理后台 CRUD | 分层 | 职责清晰、易维护 |
| 实时控制 | 闭环 / 事件驱动 | 响应快 |
| AI / 规则系统 | 黑板 / 规则引擎 | 知识分离 |
| 跨端 APP | C/S 或 B/S | 部署灵活 |

## 三、题型 2：质量属性识别题（ATAM 四类点）

> 典型问法：从以下架构决策中找出敏感点、权衡点、风险点、非风险点

**套路**：表格化答题，**每条决策 → 判类型 → 写理由**

| 决策 | 类型 | 理由（质量属性关联） |
|---|---|---|
| 主备切换 <30s | 敏感点 | 影响**可用性** |
| 使用 HTTPS | 权衡点 | +安全 / -性能 |
| MQ 未持久化 | 风险点 | 宕机丢消息，影响可靠性 |
| 采用 Kafka | 非风险点 | 业界成熟 |

**识别信号词**：

- "为了…采用…" → 敏感点
- "既影响 X 又影响 Y" → 权衡点
- "可能…将会…" → 风险点
- "已经验证 / 成熟" → 非风险点

## 四、题型 3：数据库设计题

> 典型问法：给 E-R 图，转关系模式 + 判范式 + 分解到 3NF

**套路**：

1. **列出关系模式**：`R(A, B, C, D, …)`
2. **给出函数依赖集**：`F = {A→B, B→C, …}`
3. **求候选码**：从 L（只在左出现）属性开始求闭包
4. **判范式**：1NF → 2NF → 3NF 逐级验证
5. **分解**：
   - 2NF 违反 → 分出部分依赖
   - 3NF 违反 → 分出传递依赖
6. **验证**：无损连接 + 依赖保持

## 五、题型 4：层次式架构 / 信息系统架构

> 典型问法：设计某系统架构，画图 + 说明

**答题 6 要素**：

1. **分层图**（4 层经典：表现 / 业务 / 数据访问 / 数据）
2. **每层职责**
3. **关键组件**（网关、缓存、MQ、DB）
4. **数据流**
5. **质量属性保障**
6. **关键技术选型 + 理由**

## 六、题型 5：微服务 / 云原生改造

> 典型问法：单体如何拆分微服务？

**6 步法**：

1. **识别限界上下文**（DDD 领域建模）
2. **按业务能力拆分**（订单 / 支付 / 用户 / 库存）
3. **数据库拆分**（Database Per Service）
4. **通信机制**（同步 REST/gRPC + 异步 MQ）
5. **基础设施**（注册中心、网关、链路追踪、配置中心）
6. **治理**（熔断、降级、限流、服务网格）

## 七、题型 6：分布式事务

**给场景选方案**：

| 场景 | 推荐 | 关键理由 |
|---|---|---|
| 金融转账 | TCC | 强一致 |
| 电商下单 | 本地消息表 / MQ 事务 | 最终一致 + 高性能 |
| 库存扣减 | Redis + Lua / 异步对账 | 高并发 |
| 跨库统计 | XA / 2PC | 简单 |

## 八、题型 7：安全架构

**纵深防御 5 层答题**：

1. **网络层**：防火墙、WAF、DDoS 清洗
2. **主机层**：HIDS、漏洞扫描、基线加固
3. **应用层**：SSO、MFA、RBAC、输入校验、CSP
4. **数据层**：加密（传输 TLS、存储 AES、字段脱敏）
5. **审计**：全链路日志、SIEM、合规报告

### STRIDE 威胁识别

| 威胁 | 缓解 |
|---|---|
| **S**poofing 仿冒 | 认证 |
| **T**ampering 篡改 | 完整性校验（HMAC） |
| **R**epudiation 抵赖 | 日志 + 签名 |
| **I**nformation Disclosure 泄露 | 加密 |
| **D**oS 拒服 | 限流 + 清洗 |
| **E**levation of Privilege 提权 | 最小权限 |

## 九、题型 8：大数据架构

**Lambda vs Kappa 选择**：

| 场景 | 选 | 理由 |
|---|---|---|
| 需要精确批处理结果 | Lambda | 有专门批处理层 |
| 只需流处理 | Kappa | 简化运维 |

## 十一、填表题套路

当题目给空白表格，按**题干关键词**抽出**维度**填：

- 功能 → 职责 / 输入 / 输出 / 相关模块
- 质量属性 → 指标 / 战术 / 影响
- 技术选型 → 方案 / 优点 / 缺点 / 适用场景

## 十二、万能兜底句式（卡壳时用）

- "该决策体现了 **XX 战术**，具体表现为…"
- "为达到 **可用性 99.9%**，采取了 **主备 + 心跳 + 自动切换** 三重保障"
- "通过 **解耦 / 缓存 / 异步** 提升性能"
- "权衡考虑了 **成本、复杂度、团队熟悉度**"
