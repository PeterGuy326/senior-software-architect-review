# ABSD / ADL / ISO 42010 架构设计方法速查（📚🎯 综合 + 案例高频）

## 一、ABSD（Architecture-Based Software Design）

**基于架构的软件设计方法**，是系统架构设计师教程第 7 章重点。

### 三个基础

1. **功能分解**
2. **通过选择架构风格实现质量和业务需求**
3. **软件模板的使用**

### 视角 / 视图（多视图方法）

- **概念视图**：系统主要抽象、构件
- **模块视图**：代码模块组织
- **执行视图**：运行时动态行为
- **代码视图**：最终实现

### ABSD 过程（6 步）

1. **架构需求**：需求获取 → 标识构件 → 需求评审
2. **架构设计**：提出架构模型 → 映射构件 → 分析构件交互 → 生成架构 → 评审
3. **架构文档化**
4. **架构复审**
5. **架构实现**
6. **架构演化**

## 二、ADL（Architecture Description Language）

**架构描述语言**：形式化描述架构

### 典型 ADL

| ADL | 特点 |
|---|---|
| **UniCon** | 卡梅隆大学，支持多风格 |
| **Wright** | 使用 CSP 描述交互 |
| **Rapide** | 事件驱动，支持可执行模拟 |
| **Darwin** | 分布式系统 |
| **Acme** | 通用交换语言 |
| **C2** | 层次式、消息驱动 |

### ADL 核心元素

- **构件**（Component）
- **连接件**（Connector）
- **架构配置**（Configuration）

## 三、ISO/IEC/IEEE 42010 架构描述标准

### 核心概念

```
系统 → 由利益相关者关注 → 通过架构视角观察 → 形成架构视图
```

### 关键术语

| 术语 | 含义 |
|---|---|
| **Stakeholder** 利益相关者 | 用户、架构师、客户、运维等 |
| **Concern** 关注点 | 某一利益相关者关心的议题 |
| **Viewpoint** 视角 | 解决关注点的模式（模板） |
| **View** 视图 | 应用视角到架构得到的结果 |
| **Architecture Description** 架构描述 | 所有视图的集合 |

## 四、Kruchten 4+1 视图模型（与 42010 兼容）

| 视图 | 关注点 | 典型 UML |
|---|---|---|
| **逻辑视图** | 功能、结构 | 类图、对象图、状态图 |
| **开发视图** | 代码组织、模块 | 组件图、包图 |
| **进程视图** | 并发、性能、可靠性 | 活动图、顺序图 |
| **物理视图** | 部署、硬件拓扑 | 部署图 |
| **+1 场景视图** | 用例，串联以上四个 | 用例图 |

## 五、软件产品线（SPL）

### 两个生命周期

1. **领域工程**：识别共性 + 可变性 → 构建核心资产
2. **应用工程**：利用核心资产快速开发单个产品

### 三角色

- 领域专家
- 领域分析师
- 领域设计师

### 典型方法：**DSSA**（Domain-Specific Software Architecture）

### 产品线成功要素

- 核心资产的可复用性
- 变化性管理
- 组织结构适配（领域团队 + 应用团队）

## 六、ABSD vs 传统设计方法对比

| 维度 | 传统 OO | **ABSD** |
|---|---|---|
| 驱动 | 功能需求 | **质量属性 + 业务需求** |
| 产出 | 类图 | **架构视图集** |
| 重用粒度 | 类 / 模块 | **架构 / 产品线** |

## 七、答题套路（案例题）

当题目问"如何进行架构设计"，按 ABSD 6 步走：

1. 架构需求（**识别关键质量属性**）
2. 选择架构风格（对照业务特征）
3. 构件与连接件设计
4. 架构文档化（4+1 视图）
5. 架构复审（**用 ATAM**）
6. 架构实现 + 演化
