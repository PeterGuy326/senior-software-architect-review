# 范文 01 · 论软件架构设计及其应用

> **题目**：论软件架构设计及其应用
> **案例改编**：某电商平台订单中台建设项目（2023-2024）
> **字数**：约 2800 字

---

## 摘要（约 320 字）

2023 年 3 月，我所在公司启动了"订单中台"建设项目，目标是整合集团下 B2C 商城、跨境电商、线下门店三条业务线原本各自独立的订单系统，支撑日订单 **800 万笔**、大促峰值 **QPS 5 万**的业务规模。我作为该项目的**系统架构设计师**，主导了整体架构方案的制定与落地。项目历时 **10 个月**，团队规模 **25 人**。

本文结合该项目，论述软件架构设计的过程与应用。首先介绍项目背景与关键质量属性，然后阐述软件架构风格与多视图建模的核心理论，最后重点展开架构设计的四个关键环节：**架构风格选型**、**构件划分**、**多视图建模**、**ATAM 架构评估**。项目于 **2024 年 1 月**顺利上线，核心指标全部达标：**订单下单 P99 延迟 180ms**、**可用性 99.99%**、**大促零 P0 故障**，新业务接入周期从 **3 个月缩短到 2 周**。项目沉淀的架构经验也被复用到后续多个中台项目。

---

## 一、项目背景（约 350 字）

我所在公司是一家年销售额百亿级的综合零售集团。集团下三条业务线——B2C 商城（十年老系统，单体 Java 架构）、跨境电商（外采 SaaS）、线下门店（传统 ERP）——在订单处理上**高度分裂**：数据模型不一致、促销规则重复开发、跨业务线库存共享困难。2023 年初，业务部门提出打通三线订单、支撑**全渠道一盘货**的战略诉求。

项目启动时，我作为系统架构设计师主导设计工作。关键质量属性包括：**性能**（下单 P99 < 200ms、大促 QPS 5 万）、**可用性**（99.99%，全年停机 < 52.6 分钟）、**可修改性**（新业务线接入 < 1 个月）、**一致性**（订单与库存数据最终一致，核心支付强一致）、**安全性**（符合等保三级、支付 PCI-DSS 合规）。团队由 25 人组成，包括架构师 2 人、后端 15 人、前端 4 人、测试 3 人、运维 1 人，计划 10 个月交付。

## 二、核心理论（约 360 字）

软件架构是系统的**基本结构**，包括软件元素、元素之间的关系以及二者的特性。良好的架构设计是系统满足非功能需求的前提。

**主流架构风格**可归纳为五类：数据流（管道—过滤器）、调用—返回（分层、OO、C/S）、独立构件（事件驱动、发布—订阅）、虚拟机（解释器）、数据中心（黑板、数据仓库）。实践中往往**混合采用**，如"分层 + 事件驱动 + 微服务"。

**4+1 视图模型**由 Kruchten 提出，从**逻辑、开发、进程、物理**四个视图 + **场景视图**串联，对架构进行多角度建模，避免单视图遗漏。

**ABSD（基于架构的软件开发）**过程包含六步：架构需求、架构设计、文档化、架构复审、架构实现、架构演化。其中架构复审常采用 **ATAM** 方法，通过构建**质量属性效用树**和**场景集**，识别**敏感点、权衡点、风险点、非风险点**四类关键架构决策。

上述理论构成了我在订单中台项目中设计决策的方法论基础。

## 三、实践论述（约 1600 字）

### 3.1 架构风格选型

面对 QPS 5 万、跨业务线集成、未来可扩展的多重需求，我组织架构评审会议讨论了三种候选方案：**单体分层、SOA + ESB、微服务 + 事件驱动**。综合评估后选择 **"微服务 + 事件驱动"混合风格**。理由有三：其一，微服务的**独立部署**支持三条业务线**独立迭代**；其二，**事件驱动**通过消息队列实现跨服务**松耦合**，避免老系统改造时形成级联故障；其三，**分层风格**在服务内部保留，降低开发门槛。

同时我们**保留一小部分单体**：促销引擎因逻辑密集、事务边界强，继续保留为单体模块，与微服务通过 RPC 交互。这种"微服务 + 有选择保留单体"的决策，避免了"微服务一把梭"的过度工程。

### 3.2 构件划分与连接

基于 **DDD 限界上下文**，我将订单中台划分为 **10 个业务服务 + 5 个基础服务**：业务侧包括订单、支付、履约、促销、库存、会员、商品、结算、售后、对账；基础侧包括 API 网关（Spring Cloud Gateway）、注册中心（Nacos）、配置中心（Apollo）、链路追踪（SkyWalking）、统一日志（ELK）。

服务间**同步通信**使用 gRPC（内部高性能）和 REST（对外兼容），**异步通信**使用 RocketMQ（金融级可靠）。每个服务独立数据库（**Database Per Service** 红线）。关键数据通过 Canal 监听 MySQL binlog，实现 ES 搜索索引和缓存的**准实时同步**。

### 3.3 多视图建模

我采用 **4+1 视图模型**进行多角度建模，重点展开三个视图：

**逻辑视图**使用 UML 类图刻画领域模型，明确订单聚合根、支付聚合根、履约聚合根之间的边界；**进程视图**使用时序图描述"下单 → 锁库存 → 创建支付单 → 发送履约事件"关键链路，标注每步的**同步/异步**特性；**物理/部署视图**绘制 K8s 拓扑，包含 3 个可用区 × 30 节点，核心服务各 6 副本跨区部署。

开发视图主要用于指导代码结构（Maven 多模块 + Git 分支策略），场景视图串联用例。在**架构文档化**阶段，我用 PlantUML 维护这些视图，并将文档提交到 Git，保证"**图与代码同源**"。

### 3.4 架构评估与改进

采用 **ATAM 方法**进行架构复审。首先构建**效用树**，将性能、可用性、可修改性、安全性作为四个二级节点，每个节点下挂具体场景。针对最高优先级的"性能 - 下单 P99 < 200ms (H,H)"场景，分析发现多级缓存策略是敏感点。

评审识别出 **4 个敏感点**（主备切换 < 30s、RocketMQ 持久化、下单 TCC 补偿、API 限流）、**2 个权衡点**（全链路 HTTPS 安全↑性能↓、异步扣库存性能↑一致性↓）、**3 个风险点**（促销引擎单点无熔断、对账服务未做压测、DR 演练频次不足）、**5 个非风险点**（Nacos 集群、MySQL 主从半同步、Redis 哨兵、K8s 滚动发布、SkyWalking）。

针对每个风险点，团队落实了改进措施：促销引擎引入 Sentinel 熔断并拆分只读副本；对账服务补做全链路压测；DR 演练从半年一次提升至每季度一次。上线前的第二次复审确认风险点**全部转为非风险点**。

## 四、总结与展望（约 260 字）

项目于 2024 年 1 月按期上线，并经受了 2024 年 "6·18" 和 "双 11" 两次大促考验：峰值 **QPS 达 6.2 万**、**下单 P99 延迟 178ms**、**全年可用性 99.995%**、**零 P0 故障**，核心质量目标**全部达成**。新业务线接入周期从 3 个月缩短到 2 周，显著提升了中台的**杠杆价值**。

回顾项目，我体会到架构设计的关键在于**权衡**而非教条：既要识别关键质量属性并选型对应风格，也要敢于**保留合理的单体**；既要分层解耦，也要用 ATAM 闭环验证决策。

不足之处在于**治理成本**高于预期：15 个服务的监控告警、链路追踪、版本灰度等运维工作量大，后续计划引入 Istio 服务网格进一步下沉治理能力，并通过 GitOps + Argo CD 提升变更效率。
