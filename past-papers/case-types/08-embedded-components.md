# 案例题型 08 · 嵌入式 / 构件组装

> 中频题型，约 2–3 年一次，25 分。

## 考点分布

- 嵌入式**实时系统**设计
- **构件**（Component）识别、接口契约
- 嵌入式软件架构风格（分层、事件驱动、任务调度）
- 实时调度算法（RM / EDF）
- 与硬件的接口设计

## 嵌入式核心理论

### 嵌入式软件分类

| 类型 | 特点 | 例子 |
|---|---|---|
| **硬实时** | 违反时限 = 系统失败 | 航天、汽车 ABS |
| **软实时** | 违反时限 = 降级 | 音视频、IoT |
| **弱实时** | 不严格 | 家电 |

### 嵌入式架构风格

1. **分层架构**：硬件抽象层 HAL → 驱动 → 中间件 → 应用
2. **事件驱动**：中断 + 事件循环
3. **任务调度**：RTOS 多任务 + 优先级
4. **管道-过滤器**：信号处理链

### 实时调度算法

| 算法 | 特点 |
|---|---|
| **RMS**（Rate Monotonic） | 频率越高优先级越高，**静态** |
| **EDF**（Earliest Deadline First） | 截止期最近优先，**动态** |
| **DMS** | 截止期越短优先级越高 |

### 可调度判定

- **RMS**：Σ(Ci/Ti) ≤ n(2^(1/n) - 1)，n→∞ 时极限 **0.693**
- **EDF**：Σ(Ci/Ti) ≤ 1（利用率满 100% 可达）

## 构件化核心理论

### 构件要素

- **接口契约**（Interface）
- **独立部署**
- **无持久状态**（或明示）
- **可组装**

### 主流标准

- **CORBA**（跨语言）
- **COM/DCOM**（微软）
- **EJB**（Java EE）
- **OSGi**（模块化 Java，支持热插拔）

### 构件复用层次

1. 代码复用（库）
2. 设计复用（模式）
3. 分析复用（领域模型）
4. 架构复用（参考架构）

## 答题模板

### 问题 1：嵌入式架构设计

**模板**：
```
1. 分层：HAL → BSP → RTOS → 中间件 → 应用
2. 调度：FreeRTOS / μC/OS，RMS 静态优先级
3. 通信：消息队列 / 信号量 / 共享内存
4. 实时保证：中断响应 < 10μs，关键任务周期 10ms
5. 可靠性：看门狗 WDT + 双机热备
```

### 问题 2：可调度分析

**套路**：
```
给定任务周期 Ti 和执行时间 Ci：
1. 计算 U = Σ(Ci/Ti)
2. 对比 RMS 上界 n(2^(1/n)-1)
3. U ≤ 上界 → 可调度；否则必须用 EDF 或优化
```

### 问题 3：构件识别与组装

**套路**：
1. 识别可复用单元（UI / 通信 / 数据处理 / 算法）
2. 定义接口契约（如 OpenAPI / proto）
3. 组装：依赖注入 / IoC 容器
4. 版本管理：语义化版本

### 问题 4：嵌入式 + 物联网

**模板**：
```
端（MCU + RTOS）→ 边（网关 Gateway）→ 云（IoT 平台）
- 端：分层 + 事件驱动
- 边：数据过滤 + 协议转换（MQTT / CoAP）
- 云：设备管理 + 规则引擎 + 大数据
```

## 万能高分句

- "采用**分层架构**：**HAL → BSP → RTOS（FreeRTOS）→ 中间件 → 应用**，实现**硬件解耦**与**可移植**"
- "关键任务使用 **RMS 静态优先级**调度，经可调度性分析 **U = 0.65 < 0.693**，满足实时约束"
- "通过**看门狗 WDT**  + **双机冷备 / 热备**保障 **99.999%** 可用"
- "构件接口以 **OpenAPI** 契约先行，通过 **OSGi** 支持运行时**热插拔**"

## 常见陷阱

| ❌ | ✅ |
|---|---|
| 不区分硬实时/软实时 | 题干要看清**失败后果** |
| 可调度分析没计算 | **列出公式 + 代入数字** |
| 构件 = 类 | 构件是**可独立部署**的单元 |
| 忽视硬件 | 嵌入式必须讲 **HAL / BSP / 中断** |
| 无可靠性手段 | **看门狗 / 双机 / ECC** 必写 |
