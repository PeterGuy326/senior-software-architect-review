# 论文主题 10 · 设计模式在架构中的应用

## 历年出题角度

- 论软件设计模式及其应用
- 论设计模式在架构中的应用
- 论面向对象的设计原则

## 一、核心理论

### OO 设计原则（SOLID + 其他）

| 原则 | 含义 |
|---|---|
| **S**RP | 单一职责 |
| **O**CP | 开闭原则（开放扩展、关闭修改） |
| **L**SP | 里氏替换 |
| **I**SP | 接口隔离 |
| **D**IP | 依赖倒置 |
| 迪米特 | 最少知识 |
| 合成复用 | 优先组合而非继承 |

### GoF 23 种模式分类

- **创建型（5）**：单例 / 工厂方法 / 抽象工厂 / 建造者 / 原型
- **结构型（7）**：适配器 / 桥接 / 组合 / 装饰 / 外观 / 享元 / 代理
- **行为型（11）**：责任链 / 命令 / 解释器 / 迭代器 / 中介者 / 备忘录 / 观察者 / 状态 / 策略 / 模板方法 / 访问者

### 架构级模式（比 GoF 高一层）

- **MVC** / MVVM / MVP
- **分层** / 六边形 / 洋葱
- **CQRS**（读写分离）
- **事件溯源**（Event Sourcing）
- **微服务模式**（Sidecar、BFF、Saga）

## 二、万能提纲

```
1. 背景（350 字）
   - 【电商订单 / 支付核心】系统，频繁变更、扩展难
2. 理论（350 字）
   - SOLID、GoF 三大类、架构级模式
3. 实践论述（1600 字）⭐⭐⭐⭐⭐
   (1) 策略模式：支付方式（微信 / 支付宝 / 银联）可插拔
   (2) 工厂 + 抽象工厂：多数据源（MySQL/Oracle）切换
   (3) 责任链：订单校验（库存 / 价格 / 风控）
   (4) 观察者：订单状态变更 → 消息通知
   (5) 装饰器：订单优惠券叠加（满减 / 折扣 / 秒杀）
   (6) CQRS：读走 ES，写走 MySQL
4. 成效（250 字）
   - 新支付方式接入 3d → 0.5d
```

## 三、关键数据

- 模式落地：**5–8 种 GoF + 2 种架构级**
- 扩展成本：**降低 70%**

## 四、万能句式

- "新增支付渠道采用**策略模式**，符合 **OCP** 开闭原则，无需改调用方"
- "订单校验链通过**责任链模式**实现，每个校验器单一职责（SRP）"
- "读写分离采用 **CQRS** 架构模式，读用 ES 写用 MySQL"

## 五、避坑点

| ❌ | ✅ |
|---|---|
| 罗列 23 种模式 | 选 **5–8 种** 深入讲 + 代码片段级细节 |
| 模式名乱贴 | **场景 + 目的 + 具体类设计** |
| 只谈 GoF | 加 **架构级模式**（CQRS / 事件溯源） |
| 不讲权衡 | 每种模式都写**为什么选它而不是 XX** |
