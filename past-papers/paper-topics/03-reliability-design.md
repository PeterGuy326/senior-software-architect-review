# 论文主题 03 · 系统可靠性设计

## 历年出题角度

- 论系统可靠性分析与设计（**高频经典**）
- 论软件可靠性设计与实现
- 论软件容错技术及其应用

## 一、核心理论

### 可靠性指标（必写）

| 指标 | 含义 | 计算 |
|---|---|---|
| **MTBF** | 平均无故障时间（Mean Time Between Failures） | — |
| **MTTR** | 平均修复时间（Mean Time To Repair） | — |
| **可用性 A** | Availability | A = MTBF / (MTBF + MTTR) |
| **SLA** | 服务级别协议 | 99.9% / 99.99% / 99.999% |

### 可用性对应时间（金融必备）

| 可用性 | 年停机 |
|---|---|
| 99% | 3.65 天 |
| 99.9% | 8.76 小时 |
| **99.99%** | **52.6 分钟** |
| **99.999%** | **5.26 分钟**（电信级） |

### 容错技术

| 技术 | 原理 |
|---|---|
| **N 版本程序设计** | 独立开发 N 份，多数表决 |
| **恢复块** | 主块失败切备份块 |
| **防卫式程序** | 异常检查、assert |

### 冗余分类

- **结构冗余**：静态（主备）/ 动态（双机热备）/ 混合
- **信息冗余**：校验码、CRC
- **时间冗余**：重试、补偿
- **冗余附加**：额外硬件 / 软件

## 二、万能提纲（金融 / 电信级案例）

```
1. 项目背景（350 字）
   - 【金融支付 / 电信计费】系统，可用性要求 99.99%+
2. 理论（350 字）
   - MTBF / MTTR / A 公式、冗余分类、容错技术
3. 实践论述（1600 字）⭐⭐⭐⭐⭐
   (1) 故障检测：心跳 + 健康检查 + 监控告警（Prometheus）
   (2) 冗余设计：同城双活 + 异地灾备（三地五中心）
   (3) 自动故障切换：Keepalived / 哨兵 / K8s Liveness
   (4) 流量保护：熔断（Hystrix/Sentinel）+ 限流 + 降级
   (5) 数据可靠性：多副本同步 / 异步 + 事务消息
4. 总结（200 字）
   - 可用性从 99.9% → 99.99%
   - 全年未发生 P0 级事故
```

## 三、关键数据（填入即可）

- MTBF：目标 **30 天以上**
- MTTR：目标 **< 5 分钟**
- 可用性：**99.99%**（四个 9）
- 年停机：**< 52.6 分钟**
- RPO（数据恢复点目标）：**< 5s**
- RTO（恢复时间目标）：**< 30s**

## 四、常用技术栈（论文里写具体）

- 负载均衡：Nginx / LVS / F5
- 服务注册：Nacos / Consul
- 熔断降级：Sentinel / Hystrix / Resilience4j
- 消息队列：**RocketMQ（金融级）**
- 数据库：MySQL 主从半同步 + MHA / Orchestrator
- 缓存：Redis Cluster 主从 + 哨兵
- 监控：Prometheus + Grafana + Alertmanager

## 五、万能句式

- "针对金融级可用性要求（**99.99% SLA**），我们从**故障预防、检测、恢复**三个维度设计"
- "通过**同城双活 + 异地灾备**架构，实现 **RPO < 5s，RTO < 30s**"
- "上线后**连续 12 个月**保持 99.99%+ 可用性，全年无 P0 故障"

## 六、避坑点

| ❌ | ✅ |
|---|---|
| 没有量化指标 | 必须写 SLA / MTBF / 具体分钟数 |
| 只讲硬件冗余 | **三维度齐全**：结构 + 信息 + 时间 |
| 不提监控 | 监控告警是故障检测的基础，必写 |
| 不提应急预案 | 补上**故障演练 / 混沌工程** |
