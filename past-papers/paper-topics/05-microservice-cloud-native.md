# 论文主题 05 · 微服务与云原生架构

## 历年出题角度

- 论微服务架构设计及其应用
- 论无服务器架构（Serverless）
- 论云原生架构在 XX 中的应用
- 论基于容器的应用架构

## 一、核心理论

### 云原生四要素

1. **容器化**（Docker）
2. **微服务**
3. **DevOps**
4. **持续交付**

### 十二要素应用（12-Factor）

代码库、依赖、配置、后端服务、构建/发布/运行、进程、端口绑定、并发、易处理、开发/生产一致、日志、管理进程

### 微服务拆分原则（DDD）

- **单一职责**
- **限界上下文**（Bounded Context）
- **业务能力**驱动
- **团队自治**（Two-Pizza Team）
- **数据库私有**（Database Per Service）

### 核心组件

| 组件 | 代表 | 作用 |
|---|---|---|
| 注册中心 | Nacos / Eureka | 服务发现 |
| API 网关 | Spring Cloud Gateway / Kong | 统一入口 |
| 配置中心 | Apollo / Nacos | 配置管理 |
| 熔断限流 | Sentinel / Hystrix | 治理 |
| 链路追踪 | SkyWalking / Jaeger | 可观测 |
| 服务网格 | Istio / Envoy | 流量治理 |
| 编排 | **Kubernetes** | 容器调度 |

## 二、万能提纲（单体改造微服务）

```
1. 背景（400 字）
   - 原单体系统瓶颈：耦合高、发布慢、性能差
   - 改造目标：QPS 从 5k → 50k，部署频率日级 → 小时级
2. 理论（350 字）
   - 云原生、12-Factor、微服务拆分原则
3. 实践论述（1600 字）⭐⭐⭐⭐⭐
   (1) 服务拆分：DDD 限界上下文（订单/支付/库存/用户/商品）
   (2) 数据库拆分：Database Per Service + 分库分表
   (3) 通信：同步 REST/gRPC + 异步 MQ（RocketMQ）
   (4) 基础设施：K8s + Istio + Prometheus
   (5) 分布式事务：TCC（核心交易）+ 本地消息表（最终一致）
   (6) 服务治理：熔断降级限流（Sentinel）+ 链路追踪（SkyWalking）
4. 成效（250 字）
   - QPS 10 倍、发布频率 24 倍、可用性 99.99%
```

## 三、关键数据

- QPS：峰值 **10 万**
- 部署频率：**每天多次**（从月级提升）
- 服务数：**50–100 个**
- 容器数：**500–1000**（K8s 集群 30 节点）
- MTTR：**从小时级降到分钟级**

## 四、万能句式

- "通过 **DDD 限界上下文** 将单体拆分为 10 个业务微服务 + 5 个基础服务"
- "核心交易采用 **TCC** 保证强一致，辅助场景用 **本地消息表** 实现最终一致"
- "全部容器化，通过 **Kubernetes** 编排，**Istio** 做服务网格治理"
- "**Serverless（FaaS）** 用于非核心异步场景（如图片处理），按需付费，成本降低 **40%**"

## 五、避坑点

| ❌ | ✅ |
|---|---|
| "全部微服务化" | **共同数据** / **紧耦合**场景留在单体 |
| 不谈治理 | 熔断限流链路追踪必写 |
| 忽视分布式事务 | 选**合适方案**：TCC / Saga / MQ |
| 数据库不拆 | Database Per Service 是红线 |
| 无监控 | 可观测三支柱：Metrics / Logging / Tracing |
