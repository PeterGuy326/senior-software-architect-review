# 论文主题 09 · 软件架构演化与维护

## 历年出题角度

- 论软件系统架构的演化（**高频**）
- 论软件架构的维护与演化
- 论遗留系统的改造

## 一、核心理论

### 架构演化的驱动力

- **业务变化**（新需求、规模膨胀）
- **技术变化**（新技术、技术债）
- **非功能需求**（性能、可用性、安全）
- **组织变化**（团队结构 → 康威定律）

### 架构演化类型

| 类型 | 含义 | 代价 |
|---|---|---|
| **可修改性演化** | 小范围调整 | 低 |
| **可扩展性演化** | 新增功能 | 中 |
| **重构性演化** | 不改行为改结构 | 中 |
| **重写性演化** | 完全重写 | 高（慎用） |

### 演化模式

1. **静态演化**：停机升级，简单但有停机时间
2. **动态演化**：不停机热更新（OSGi、蓝绿、灰度）

### 康威定律

> "**组织架构决定软件架构**"

系统设计反映组织沟通结构 → 先调组织再调系统。

### 常见演化手段

- **绞杀者模式**（Strangler Fig）：逐步替换老系统
- **分支抽象**：代码级渐进重构
- **防腐层**（ACL）：新老边界隔离

## 二、万能提纲（单体→微服务渐进演化）

```
1. 背景（350 字）
   - 10 年老系统，代码 100 万行，新需求难加
2. 理论（350 字）
   - 演化驱动力、类型、康威定律、绞杀者
3. 实践论述（1600 字）⭐⭐⭐⭐⭐
   (1) 架构评估：识别痛点（耦合高、发布慢）
   (2) 演化路线图：短 3m / 中 1y / 长 3y
   (3) 绞杀者模式：API Gateway 做流量切换
   (4) 数据迁移：双写 + CDC（Canal）+ 校对
   (5) 防腐层 ACL：新服务屏蔽老系统数据模型
   (6) 灰度上线：1%→10%→50%→100%
4. 成效（250 字）
   - 迭代周期 2w → 2d，核心域解耦完毕
```

## 三、关键数据

- 演化周期：**6–24 个月**
- 代码迁移：**100w 行**
- 灰度分级：**1% / 10% / 50% / 100%**
- 零停机，**RPO=0**

## 四、万能句式

- "基于**绞杀者模式**渐进式替换，避免"大爆炸"式重写风险"
- "通过**防腐层（ACL）**隔离新老系统数据模型，保持新服务的**领域纯净**"
- "遵循**康威定律**，**先调组织**（小团队负责单服务）再调架构"

## 五、避坑点

| ❌ | ✅ |
|---|---|
| 大爆炸式重写 | **渐进式**绞杀者模式 |
| 只改技术不调组织 | 康威定律：**组织决定架构** |
| 无量化收益 | 迭代速度、故障率必须前后对比 |
| 不谈回滚 | 灰度 + 可回滚 是底线 |
