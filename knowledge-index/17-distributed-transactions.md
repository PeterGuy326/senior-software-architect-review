# 知识点 17 · 分布式事务

## 一、核心概念

### CAP / BASE

- **CAP**：一致性 / 可用性 / 分区容错（分布式必选 P，AP vs CP 二选一）
- **BASE**：**B**asically Available + **S**oft state + **E**ventually consistent

### 5 种方案对比

| 方案 | 一致性 | 性能 | 复杂度 | 适用 |
|---|---|---|---|---|
| **2PC / XA** | 强 | 差 | 中 | 传统金融、跨库 |
| **3PC** | 强 | 差 | 高 | 少见 |
| **TCC** | 强 | 中 | **高** | **核心交易** |
| **Saga** | 最终 | 好 | 中 | **长流程**（订单→物流→支付） |
| **本地消息表** | 最终 | 好 | 低 | 一般业务 |
| **MQ 事务消息** | 最终 | 好 | 低 | **RocketMQ** 原生 |

### 2PC 两阶段

1. **准备阶段**：协调者问全部参与者"能否提交"
2. **提交阶段**：全 Yes 则 commit；有 No 则 rollback

**缺点**：阻塞、单点、脑裂

### TCC 三阶段

- **Try**：预留资源（冻结库存）
- **Confirm**：确认提交（扣减冻结库存）
- **Cancel**：回滚（解冻）

**要点**：业务方**自己实现**三个方法；Confirm/Cancel **必须幂等**。

### Saga 补偿

长事务拆成 N 个子事务，每个有**正向操作**和**补偿操作**；任一失败则**反向补偿**已完成的子事务。

## 三、典型例题

### 📚 选择题 1（CAP）

MongoDB 默认是：
- A. CA  B. **AP**  C. CP  D. 三者都满足

**答案**：B（默认 AP，可配置为 CP）

### 📚 选择题 2（TCC）

TCC 的 Confirm 和 Cancel 操作**必须**满足：
- A. 原子性  B. **幂等性**  C. 持久性  D. 隔离性

**答案**：B

### 📚 选择题 3（方案选型）

下列场景最适合用 **Saga** 的是：
- A. 跨库转账  B. **订单→物流→支付→通知**（长流程）  C. 库存扣减  D. 余额查询

**答案**：B

### 🎯 案例填空：电商下单方案

| 环节 | 方案 | 理由 |
|---|---|---|
| 下单 + 扣库存 | **TCC** | 核心交易，强一致 |
| 下单 → 积分 | **本地消息表** | 弱依赖，最终一致 |
| 下单 → 物流通知 | **MQ 事务消息** | 异步解耦 |
| 多步骤复合流程 | **Saga** | 长事务 + 补偿 |

### ✍️ 论文场景

"核心支付交易采用 **TCC** 保证强一致（try 预扣 / confirm 确认 / cancel 回滚），并要求 Confirm/Cancel **幂等**；辅助通知场景使用 **本地消息表 + MQ** 实现最终一致。"
