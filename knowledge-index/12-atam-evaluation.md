# 知识点 12 · ATAM 架构评估

## 一、核心概念

### 三大评估方法

| 方法 | 特点 | 阶段 |
|---|---|---|
| **SAAM** | 基于场景，最早 | 早期 |
| **ATAM** | **关注权衡** | 设计阶段（**主流**） |
| **CBAM** | ATAM + 经济学 | 投资决策 |

### ATAM 四类关键点（**必考**）

| 类型 | 信号词 | 例子 |
|---|---|---|
| **敏感点** | "为了 XX，我们选..." | 主备切换 < 30s 对可用性 |
| **权衡点** | "既...又...（有正有负）" | HTTPS 提升安全但降性能 |
| **风险点** | "可能导致 / 尚未验证" | 单点 MQ 未持久化 |
| **非风险点** | "已验证 / 业界成熟" | Kafka 集群、MySQL 主从 |

### ATAM 9 步流程

1. 介绍 ATAM
2. 业务驱动
3. 被评估架构
4. 识别架构方法
5. **质量属性效用树**（核心）
6. 分析架构方法
7. 场景集 + 优先级
8. 分析架构方法（迭代）
9. 呈现结果

### 效用树优先级

用 **(业务价值, 技术难度)** 标注，取值 **H / M / L**

## 三、典型例题

### 📚 选择题 1（评估方法）

ATAM 最独特的贡献是：
- A. 识别敏感点  B. 构建效用树  C. **关注质量属性之间的权衡**  D. 成本分析

**答案**：C

### 📚 选择题 2（四类点判定）

"全链路 HTTPS 加密既提升安全又降低约 10% 性能"属于：
- A. 敏感点  B. **权衡点**  C. 风险点  D. 非风险点

**答案**：B

### 🎯 案例填空：四类点判定

依次识别以下决策：

| 决策 | 类型 | 理由 |
|---|---|---|
| 主备切换时间 < 30s | **敏感点** | 直接决定可用性目标达成 |
| 异步写入 DB 提升性能但可能数据丢失 | **权衡点** | 性能↑ 一致性↓ |
| MQ 集群未做持久化 | **风险点** | 宕机可能丢消息，未验证 |
| 采用 Kafka 做消息中间件 | **非风险点** | 业界成熟、已广泛验证 |

### ✍️ 论文场景

"通过 **ATAM** 评估，识别出 **3 个敏感点、2 个权衡点、4 个风险点**；针对风险点【单点 MQ】引入【RocketMQ 主从 + 持久化】战术，**将风险转化为非风险**。"
